# 5 ПРОГРАММА И МЕТОДИКА ИСПЫТАНИЙ

Испытания программного средства проводятся для проверки соответствия разработанной веб-платформы функциональным требованиям, выявления ошибок в работе клиентской и серверной частей, проверки корректности взаимодействия с базой данных и подтверждения возможности использования системы в основных пользовательских сценариях.

Разрабатываемая веб-платформа предназначена для подбора туристических маршрутов с персональными рекомендациями. Поэтому в ходе испытаний необходимо проверить не только отдельные программные функции, но и полный пользовательский сценарий: регистрацию, вход в систему, настройку профиля, получение рекомендаций, выбор точек на карте, сохранение маршрута и повторное открытие сохранённых данных.

В ходе проверки особое внимание уделялось запуску клиентской и серверной частей, подключению к базе данных PostgreSQL, наличию тестовых туристических объектов, сборке проекта, проверке зависимостей, доступности серверного API, регистрации и авторизации пользователя, получению туристических объектов, формированию рекомендаций, а также построению и сохранению маршрута.

## 5.1 Объект испытаний

Объектом испытаний является веб-платформа для подбора туристических маршрутов с персональными рекомендациями. Система включает клиентскую часть, серверную часть и базу данных.

Клиентская часть предоставляет пользователю интерфейс для регистрации, входа, редактирования профиля, выбора туристических объектов на карте, сохранения маршрутов и просмотра ранее созданных маршрутов. Серверная часть предоставляет API для авторизации, работы с профилем, туристическими объектами, маршрутами и рекомендациями. База данных используется для хранения пользователей, туристических объектов, маршрутов, точек маршрута и активности пользователя.

Проверка приложения выполняется в локальной среде разработки. Такой вариант испытаний позволяет быстро запускать проект, отслеживать ошибки в консоли и проверять работу базы данных без размещения приложения на удалённом сервере. Локальная проверка также удобна тем, что результаты выполнения команд можно зафиксировать в виде скриншотов и использовать в пояснительной записке.

## 5.2 Условия проведения испытаний

Перед началом испытаний выполняется подготовка базы данных. Создаётся база данных, применяются SQL-скрипты структуры и начального наполнения. В базе должны присутствовать таблицы пользователей, туристических объектов, маршрутов и точек маршрута. Если структура базы данных не создана или таблицы не заполнены тестовыми данными, часть проверок будет невозможна, так как приложение не сможет отобразить объекты на карте и сформировать маршрут.

Для запуска приложения используется команда:

```bash
npm run dev
```

Данная команда запускает серверную часть и клиентскую часть одновременно. Клиентская часть доступна в браузере по адресу `http://127.0.0.1:5173`, а серверная часть принимает запросы по адресу `http://127.0.0.1:4000/api`. После выполнения команды в консоли должны появиться сообщения о запуске Vite и API-сервера. Этот вывод можно использовать как подтверждение того, что приложение запущено в локальной среде.

Для проверки production-сборки используется команда:

```bash
npm run build
```

Эта команда выполняет компиляцию серверной части TypeScript и сборку клиентского приложения. Если сборка завершается успешно, это подтверждает отсутствие критических ошибок, препятствующих подготовке приложения к размещению.

Для проведения испытаний использовались Node.js для запуска серверной части и npm-команд, React и Vite для клиентского приложения, TypeScript для проверки типов, PostgreSQL для хранения данных и браузер для проверки пользовательского интерфейса. Проверки выполнялись через браузер, PowerShell и SQL-запросы к базе данных.

## 5.3 Подготовка тестовых данных

Для проведения испытаний используются тестовые данные. Они позволяют проверить работу приложения без необходимости вручную добавлять туристические объекты через базу данных. В тестовый набор входят туристические объекты с названиями, описаниями, координатами, категориями, рейтингами, длительностью посещения и тегами.

Наличие координат позволяет проверить отображение маркеров на карте и построение линии маршрута. Наличие категорий и тегов позволяет проверить персональные рекомендации. Наличие длительности посещения позволяет проверить расчёт общего времени маршрута.

Перед запуском функциональных испытаний проверяется, что таблица туристических объектов содержит данные. Для этого можно выполнить SQL-запрос:

```sql
SELECT COUNT(*) FROM attractions;
```

Для демонстрации результата в консоли можно выполнить команду PowerShell:

```powershell
psql -tAc "SELECT COUNT(*) FROM attractions;" `
  "postgresql://postgres:7777@localhost:5432/tour_routes"
```

Ожидаемый результат - количество объектов больше нуля. Если запрос возвращает `0`, значит начальное наполнение базы данных не было выполнено, и испытания карты и рекомендаций будут некорректными.

При выполнении запроса в консоли был получен следующий результат:

```text
8
```

Полученный результат показывает, что в базе данных присутствует 8 туристических объектов. Следовательно, можно переходить к проверке карты, рекомендаций и построения маршрутов. Результат выполнения команды PowerShell можно показать на рисунке 5.1.

## 5.7 Проверка регистрации и авторизации пользователя

Регистрация и авторизация являются первыми пользовательскими сценариями. Без них пользователь не сможет сохранять маршруты и настраивать персональные рекомендации.

Проверка регистрации выполняется через форму в браузере. Пользователь вводит имя, email и пароль, после чего нажимает кнопку регистрации. Ожидаемый результат - создание нового пользователя и переход в рабочую часть приложения. Для проверки можно использовать имя `Test User`, email `test-user@example.com` и пароль `12345678`.

Регистрацию также можно продемонстрировать через API-запрос в PowerShell. Для этого выполняется команда:

```powershell
$body = @{ name="Test User"; email="test-user@example.com";
  password="12345678" } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri `
  http://127.0.0.1:4000/api/auth/register `
  -ContentType "application/json" -Body $body
```

Если пользователь уже существует, сервер вернёт сообщение об ошибке. В таком случае для скриншота можно изменить email, например использовать `test-user2@example.com`. При успешной регистрации в ответе должен быть объект пользователя и токен авторизации. Пароль в ответе отсутствует, что подтверждает корректность обработки конфиденциальных данных.

После регистрации проверяется вход в систему. Для этого пользователь выходит из приложения, открывает форму входа и вводит тот же email и пароль. Если данные корректны, приложение снова открывает рабочий интерфейс. Для демонстрации входа через консоль можно выполнить команду:

```powershell
$login = @{ email="test-user@example.com";
  password="12345678" } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri `
  http://127.0.0.1:4000/api/auth/login `
  -ContentType "application/json" -Body $login
```

Результат выполнения команды можно показать на рисунке 5.5. На скриншоте должно быть видно, что сервер возвращает данные пользователя и токен. Дополнительно проверяется негативный сценарий: вход с неправильным паролем. В этом случае сервер не должен выдавать токен, а интерфейс должен показать сообщение об ошибке.

## 5.8 Проверка загрузки туристических объектов

После входа в систему приложение должно загрузить список туристических объектов из базы данных. Эти данные используются для отображения карточек объектов и маркеров на карте.

Проверка выполняется в браузере. После авторизации пользователь переходит на вкладку карты. В левой части интерфейса должен появиться список туристических объектов, а на карте должны отобразиться маркеры. При нажатии на маркер объект добавляется в маршрут, а при выборе двух точек появляется линия маршрута.

Данную проверку можно дополнительно выполнить через API. В обычном HTTP-виде запрос выглядит следующим образом:

```http
GET /api/attractions
Authorization: Bearer <token>
```

Чтобы продемонстрировать этот запрос в консоли и сделать скриншот для пояснительной записки, сначала нужно получить токен авторизации, а затем передать его в заголовке `Authorization`. Это можно сделать в PowerShell следующими командами:

```powershell
$login = @{ email="test-user@example.com";
  password="12345678" } | ConvertTo-Json
$auth = Invoke-RestMethod -Method Post -Uri `
  http://127.0.0.1:4000/api/auth/login `
  -ContentType "application/json" -Body $login
$token = $auth.token
```

После получения токена выполняется запрос к списку туристических объектов:

```powershell
Invoke-RestMethod -Uri http://127.0.0.1:4000/api/attractions `
  -Headers @{ Authorization = "Bearer $token" } |
  Select-Object -ExpandProperty attractions |
  Select-Object name, category, rating -First 5
```

В результате в консоли должен появиться список туристических объектов с названием, категорией и рейтингом. Такой вывод удобно использовать для скриншота, потому что он наглядно показывает, что сервер возвращает данные из базы. Результат выполнения команды можно представить на рисунке 5.6.

Рисунок 5.6 - Результат получения туристических объектов через API

## 5.9 Проверка формирования рекомендаций

Рекомендации проверяются после настройки профиля пользователя. Пользователь задаёт интересы, бюджет, темп путешествия и максимальную длительность маршрута. После этого система должна сформировать рекомендуемые объекты и персональный маршрут.

Проверка через интерфейс выполняется следующим образом. Пользователь открывает вкладку профиля, выбирает интересы, например историю и природу, указывает средний бюджет и сбалансированный темп путешествия, затем сохраняет профиль и возвращается на карту. После обновления рекомендаций система должна показать рекомендованный маршрут, содержащий не менее двух точек, а также рассчитанные расстояние и длительность.

Рекомендации можно также проверить через API. Для этого используется тот же токен, который был получен при авторизации:

```powershell
Invoke-RestMethod -Uri `
  http://127.0.0.1:4000/api/recommendations `
  -Headers @{ Authorization = "Bearer $token" } |
  Select-Object -ExpandProperty generatedRoute
```

Результат выполнения команды должен содержать название рекомендованного маршрута, описание, список точек и сводку по расстоянию и времени. Этот вывод можно использовать как рисунок 5.7.

## 5.10 Проверка построения и сохранения маршрута

Основным пользовательским сценарием является построение туристического маршрута на карте. Пользователь выбирает несколько туристических объектов, после чего система рассчитывает порядок точек, расстояние и примерное время маршрута.

Проверка выполняется через интерфейс браузера. Пользователь открывает вкладку карты, выбирает два или более туристических объекта, проверяет появление линии маршрута на карте, вводит название маршрута и нажимает кнопку сохранения. После этого он переходит во вкладку сохранённых маршрутов и убеждается, что новый маршрут появился в списке.

Ожидаемый результат заключается в том, что выбранные точки отображаются в текущем маршруте, маршрут визуально отображается на карте, кнопка сохранения становится доступной только при выборе двух и более точек, а после сохранения маршрут появляется во вкладке маршрутов. Карточка маршрута должна содержать название, список точек, расстояние и длительность.

Для демонстрации сохранённых маршрутов через API сначала необходимо убедиться, что в текущем окне PowerShell уже получен токен авторизации и что у пользователя есть хотя бы один сохранённый маршрут. Если маршрут ещё не был сохранён через интерфейс, команда вернёт пустой результат, так как сервер отдаёт пустой массив маршрутов. После входа в систему и сохранения маршрута через карту можно выполнить команду:

```powershell
Invoke-RestMethod -Uri http://127.0.0.1:4000/api/routes `
  -Headers @{ Authorization = "Bearer $token" } |
  Select-Object -ExpandProperty routes |
  Select-Object name, totalDistanceKm, totalDurationMinutes
```

Если пользователь уже сохранил маршрут через интерфейс, в консоли появится список маршрутов с расстоянием и длительностью. Такой результат можно использовать как рисунок 5.8.

Рисунок 5.8 - Результат получения сохранённых маршрутов пользователя

## 5.12 Проверка обработки неверного пароля

## 5.13 Проверка построения маршрута с недостаточным количеством точек

Маршрут должен содержать не менее двух туристических объектов. Если пользователь или внешний клиент отправляет запрос только с одной точкой, сервер должен вернуть ошибку и не выполнять сохранение маршрута. Эта проверка нужна для подтверждения корректности серверной валидации.

Для демонстрации сначала можно получить один идентификатор туристического объекта:

```powershell
$one = Invoke-RestMethod -Uri http://127.0.0.1:4000/api/attractions `
  -Headers @{ Authorization = "Bearer $token" }
$id = $one.attractions[0].id
```

После этого выполняется запрос предварительного построения маршрута только с одной точкой:

```powershell
$body = @{ attractionIds=@($id); optimize=$true } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri `
  http://127.0.0.1:4000/api/routes/preview `
  -Headers @{ Authorization = "Bearer $token" } `
  -ContentType "application/json" -Body $body
```

Ожидаемый результат - ошибка, сообщающая, что для маршрута нужно выбрать минимум две точки. Данный результат можно показать на рисунке 5.11.

Рисунок 5.11 - Результат проверки маршрута с одной точкой

Такая проверка подтверждает, что серверная часть не полагается только на интерфейсные ограничения и самостоятельно контролирует корректность входных данных.

## 5.14 Проверка удаления сохранённого маршрута

После создания маршрута пользователь должен иметь возможность удалить его из личного кабинета. Данная проверка подтверждает, что удаление выполняется только для маршрутов текущего пользователя и что после удаления маршрут больше не возвращается в списке.

Для демонстрации через API можно сначала получить первый маршрут пользователя:

```powershell
$routes = Invoke-RestMethod -Uri http://127.0.0.1:4000/api/routes `
  -Headers @{ Authorization = "Bearer $token" }
$routeId = $routes.routes[0].id
```

После этого выполняется запрос удаления:

```powershell
Invoke-RestMethod -Method Delete -Uri `
  "http://127.0.0.1:4000/api/routes/$routeId" `
  -Headers @{ Authorization = "Bearer $token" }
```

Если команда выполнилась без вывода ошибки, маршрут был удалён. После этого можно повторно запросить список маршрутов и убедиться, что удалённый маршрут исчез. Результат проверки можно представить на рисунке 5.12.
