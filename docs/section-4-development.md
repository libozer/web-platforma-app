# 4 Разработка программных модулей

В данном разделе рассматривается разработка программных модулей веб-платформы для подбора туристических маршрутов с персональными рекомендациями. На этапе реализации основное внимание уделялось созданию такой структуры приложения, при которой отдельные части системы выполняют строго определённые задачи и могут развиваться независимо друг от друга. Это особенно важно для веб-платформы, так как она объединяет несколько разных по назначению функций: регистрацию пользователей, хранение профиля, работу с туристическими объектами, построение маршрутов, расчёт рекомендаций и отображение интерактивной карты.

Разрабатываемая система построена по клиент-серверной архитектуре. Клиентская часть отвечает за взаимодействие с пользователем и отображение данных. Серверная часть принимает запросы от клиента, выполняет проверку данных, обращается к базе данных, рассчитывает рекомендации и возвращает результат в формате JSON. В качестве системы управления базами данных используется PostgreSQL, что позволяет хранить структурированные данные о пользователях, маршрутах, достопримечательностях и действиях пользователя.

В ходе разработки были выделены следующие группы программных модулей:

- модуль подключения к базе данных;
- модуль описания типов данных;
- модуль регистрации и авторизации;
- модуль управления пользователями и профилем;
- модуль работы с туристическими объектами;
- модуль пользовательской активности;
- модуль расчёта расстояний и оптимизации порядка точек;
- модуль построения и сохранения маршрутов;
- модуль персональных рекомендаций;
- модуль обработки HTTP-запросов;
- модуль клиентского API;
- модуль общего состояния приложения;
- модуль авторизационного интерфейса;
- модуль карты и конструктора маршрута;
- модуль профиля пользователя;
- модуль сохранённых маршрутов.

Такое разделение позволяет избежать ситуации, когда вся логика приложения находится в одном большом файле. Каждый модуль решает отдельную задачу: одни модули работают с базой данных, другие отвечают за бизнес-логику, третьи формируют пользовательский интерфейс. В результате код становится проще для тестирования, изменения и расширения.

## 4.1 Модуль подключения к базе данных

Модуль подключения к базе данных является базовым серверным модулем, так как почти все функции платформы используют постоянное хранение данных. Через него выполняются запросы к PostgreSQL: создание пользователей, получение туристических объектов, сохранение маршрутов и запись пользовательской активности.

Для подключения используется пул соединений. Пул соединений позволяет не открывать новое подключение к базе данных при каждом запросе, а переиспользовать уже созданные соединения. Это повышает производительность серверной части и уменьшает нагрузку на PostgreSQL.

Фрагмент кода, отвечающий за создание пула соединений, представлен ниже.

```ts
const { Pool } = pg;

export const pool = new Pool({
  connectionString: config.databaseUrl
});
```

В данном фрагменте создаётся объект `Pool`, которому передаётся строка подключения к базе данных. Строка подключения хранится в переменной окружения. Это сделано для того, чтобы не записывать пароль и другие параметры подключения непосредственно в программный код. Такой подход удобен при переносе приложения между локальной средой, тестовым сервером и рабочим сервером.

Для выполнения запросов используется вспомогательная функция `query`.

```ts
export async function query<T extends object>(
  text: string,
  params: unknown[] = []
): Promise<DbResult<T>> {
  return pool.query(text, params) as Promise<DbResult<T>>;
}
```

Функция принимает текст SQL-запроса и массив параметров. Возвращаемый результат типизирован, поэтому TypeScript помогает контролировать структуру данных, получаемых из базы. Это снижает вероятность ошибок, связанных с неправильными названиями полей или неверными типами.

Работа модуля подключения выполняется в несколько этапов:

1. Сервер читает строку подключения из конфигурации.
2. Создаётся пул подключений к PostgreSQL.
3. Остальные модули вызывают функцию `query` или используют `pool` напрямую для транзакций.
4. После выполнения SQL-запроса результат возвращается вызывающему модулю.

Таким образом, модуль подключения изолирует работу с PostgreSQL и предоставляет единый способ выполнения запросов для всей серверной части.

## 4.2 Модуль описания структур данных

Для разработки приложения на TypeScript важно заранее описать основные структуры данных. Это позволяет использовать статическую проверку типов и быстрее находить ошибки ещё до запуска программы.

В системе используются типы пользователя, предпочтений, туристического объекта, маршрута, точки маршрута и результата рекомендаций. Например, предпочтения пользователя описывают интересы, бюджет, темп путешествия и максимальную длительность маршрута.

```ts
export interface UserPreferences {
  interests: string[];
  budget: BudgetLevel;
  pace: TravelPace;
  maxDuration: number;
}
```

Данный интерфейс используется сразу в нескольких частях системы. На сервере он нужен для сохранения и чтения профиля пользователя, а также для расчёта рекомендаций. На клиенте он используется при отображении формы профиля.

Описание туристического объекта включает данные, необходимые как для интерфейса, так и для построения маршрута.

```ts
export interface Attraction {
  id: string;
  name: string;
  city: string;
  category: string;
  description: string;
  latitude: number;
  longitude: number;
}
```

В полном варианте объект также содержит адрес, длительность посещения, бюджет, рейтинг, изображение и теги. Для построения маршрута особенно важны поля `latitude` и `longitude`, так как они используются для отображения объекта на карте и расчёта расстояний между точками.

Отдельно описывается структура маршрута. Она объединяет общие сведения и список точек.

```ts
export interface RouteWithPoints {
  id: string;
  userId: string;
  name: string;
  totalDistanceKm: number;
  totalDurationMinutes: number;
  points: RoutePoint[];
}
```

Такое описание удобно тем, что клиентская часть получает маршрут уже в готовом виде. Ей не нужно отдельно запрашивать основную запись маршрута и отдельно список туристических объектов. Сервер формирует единую структуру и отправляет её клиенту.

Использование TypeScript-интерфейсов даёт следующие преимущества:

- повышается читаемость программного кода;
- уменьшается количество ошибок при передаче данных между модулями;
- упрощается сопровождение проекта;
- становится легче расширять структуру данных в будущем;
- появляется единый формат обмена между клиентом и сервером.

## 4.3 Модуль регистрации и авторизации

Модуль регистрации и авторизации отвечает за создание учётной записи пользователя, проверку введённых данных и выдачу токена доступа. Наличие авторизации необходимо, так как маршруты и предпочтения должны быть связаны с конкретным пользователем.

Перед регистрацией данные проходят валидацию. Проверяется корректность email, минимальная длина пароля и наличие имени. Для этого используется схема валидации.

```ts
const registerSchema = z.object({
  name: z.string().trim().min(2),
  email: z.string().trim().email(),
  password: z.string().min(8),
  preferences: preferencesSchema
});
```

Эта схема описывает требования к данным регистрации. Имя должно содержать не менее двух символов, email должен соответствовать формату электронной почты, пароль должен быть не короче восьми символов. Если данные не проходят проверку, сервер возвращает ошибку и не выполняет запись в базу данных.

После проверки выполняется поиск пользователя с таким же email.

```ts
const existing = await findUserByEmail(body.email);

if (existing) {
  throw new ApiError(409, "Пользователь с таким email уже существует");
}
```

Данная проверка предотвращает регистрацию двух пользователей с одним email. Код ошибки `409` означает конфликт данных. Такая ошибка понятна клиентской части: она может показать пользователю сообщение о том, что данный email уже используется.

Если email свободен, создаётся новый пользователь.

```ts
const user = await createUser({
  name: body.name,
  email: body.email,
  passwordHash: await hashPassword(body.password),
  preferences: body.preferences
});
```

Пароль перед сохранением передаётся в функцию `hashPassword`. В базу данных сохраняется не исходный пароль, а его хеш. Это важное требование безопасности: даже если база данных окажется скомпрометирована, злоумышленник не получит пароли пользователей в открытом виде.

После успешной регистрации сервер возвращает пользователя и токен.

```ts
res.status(201).json({
  user,
  token: signAuthToken(user)
});
```

Токен используется клиентской частью при последующих запросах. Например, при получении профиля или сохранении маршрута клиент отправляет токен в заголовке `Authorization`. Сервер проверяет токен и определяет, от имени какого пользователя выполняется запрос.

Процесс регистрации можно описать по шагам:

1. Пользователь вводит имя, email и пароль.
2. Клиент отправляет данные на сервер.
3. Сервер проверяет формат данных.
4. Сервер проверяет, не занят ли email.
5. Пароль хешируется.
6. Пользователь сохраняется в базе данных.
7. Сервер формирует JWT-токен.
8. Клиент сохраняет токен и переходит в рабочую часть приложения.

Авторизация работает похожим образом, но вместо создания новой записи сервер ищет пользователя по email и сравнивает введённый пароль с сохранённым хешем. Если проверка успешна, пользователь получает новый токен доступа.

## 4.4 Модуль проверки токена

После регистрации или входа пользователь получает токен доступа. Однако сервер должен проверять этот токен при каждом защищённом запросе. Для этого используется middleware авторизации. Middleware - это промежуточная функция, которая выполняется до основного обработчика маршрута.

Сначала из HTTP-заголовка извлекается токен.

```ts
const authHeader = req.headers.authorization;
const token = authHeader?.startsWith("Bearer ")
  ? authHeader.slice("Bearer ".length)
  : null;

if (!token) {
  next(new ApiError(401, "Требуется авторизация"));
  return;
}
```

Токен передаётся в формате `Bearer <token>`. Если заголовок отсутствует или имеет неправильный формат, сервер возвращает ошибку авторизации. Это защищает закрытые маршруты от неавторизованного доступа.

Далее токен проверяется с помощью секретного ключа.

```ts
const payload = jwt.verify(token, config.jwtSecret) as {
  sub: string;
  email: string;
  role: string;
};

(req as AuthenticatedRequest).user = {
  id: payload.sub,
  email: payload.email,
  role: payload.role
};
```

Если токен действителен, из него извлекается идентификатор пользователя, email и роль. Эти данные записываются в объект запроса. После этого основной обработчик может использовать `req.user.id`, чтобы получить данные именно текущего пользователя.

Такой подход удобен тем, что логика проверки токена находится в одном месте. Обработчики профиля, маршрутов и рекомендаций не должны самостоятельно разбирать заголовок `Authorization`: они получают уже проверенного пользователя.

## 4.5 Модуль управления пользователями и профилем

Модуль управления пользователями отвечает за создание пользователя, поиск пользователя и обновление профиля. Он является промежуточным уровнем между обработчиками HTTP-запросов и базой данных.

При создании пользователя важно нормализовать предпочтения. Пользователь может не указать все параметры сразу, поэтому система должна заполнить недостающие значения по умолчанию.

```ts
export const defaultPreferences: UserPreferences = {
  interests: ["history", "culture"],
  budget: "mid",
  pace: "balanced",
  maxDuration: 360
};
```

Эти значения используются при регистрации нового пользователя. Если пользователь ещё не настроил профиль, система всё равно сможет формировать рекомендации, ориентируясь на стандартный набор интересов.

Нормализация предпочтений выполняется отдельной функцией.

```ts
return {
  interests:
    Array.isArray(value?.interests) && value.interests.length > 0
      ? value.interests
      : defaultPreferences.interests,
  budget: value?.budget ?? defaultPreferences.budget,
  pace: value?.pace ?? defaultPreferences.pace,
```

Функция проверяет, передан ли массив интересов и не является ли он пустым. Если интересы отсутствуют, используются значения по умолчанию. Аналогично заполняются бюджет, темп и максимальная длительность маршрута.

При обновлении профиля система сначала получает текущие данные пользователя.

```ts
const current = await query<UserRow>(
  "SELECT * FROM users WHERE id = $1 AND is_active = TRUE",
  [id]
);

if (!current.rows[0]) return null;
```

Эта проверка нужна для того, чтобы обновлять только существующих и активных пользователей. Если пользователь не найден, функция возвращает `null`, а обработчик API сообщает клиенту об ошибке.

После этого новые предпочтения объединяются с текущими.

```ts
const preferences = normalizePreferences({
  ...current.rows[0].preferences,
  ...input.preferences
});
```

Такой подход позволяет обновлять профиль частично. Например, пользователь может изменить только темп путешествия, не затрагивая интересы и бюджет. Система объединит старые и новые данные, после чего снова выполнит нормализацию.

Основные задачи модуля профиля:

- хранение пользовательских данных;
- защита пароля от передачи клиенту;
- поддержка стандартных предпочтений;
- частичное обновление профиля;
- подготовка данных для клиентской части;
- обеспечение связи пользователя с маршрутами и рекомендациями.

## 4.6 Модуль работы с туристическими объектами

Туристические объекты являются основой платформы. Именно из них пользователь собирает маршрут, а система формирует рекомендации. Поэтому модуль работы с туристическими объектами должен поддерживать поиск, фильтрацию и получение объектов по идентификаторам.

При запросе списка объектов формируется массив условий `where` и массив параметров `params`. Такой подход позволяет добавлять фильтры только тогда, когда они реально указаны пользователем.

```ts
const params: unknown[] = [];
const where: string[] = [];

if (filters.search) {
  params.push(`%${filters.search}%`);
  where.push(
    `(name ILIKE $${params.length} OR description ILIKE $${params.length})`
  );
}
```

Если пользователь ввёл поисковую строку, она добавляется в массив параметров. Затем создаётся условие поиска по названию и описанию. В реальной системе поиск может быть расширен полнотекстовым индексом PostgreSQL, но для дипломного проекта текущего решения достаточно, чтобы показать базовую логику поиска.

Фильтр по категории добавляется отдельно.

```ts
if (filters.category) {
  params.push(filters.category);
  where.push(`category = $${params.length}`);
}

if (filters.budgetLevel) {
  params.push(filters.budgetLevel);
  where.push(`budget_level = $${params.length}`);
}
```

Таким образом, пользователь может отобразить, например, только исторические объекты или только объекты с определённым уровнем бюджета. Все условия затем объединяются в один SQL-запрос.

```ts
const result = await query<AttractionRow>(
  `SELECT *
   FROM attractions
   ${where.length ? `WHERE ${where.join(" AND ")}` : ""}
   ORDER BY rating DESC, name ASC`,
  params
);
```

Если фильтры отсутствуют, запрос вернёт все туристические объекты. Если фильтры есть, они объединяются оператором `AND`. Результат сортируется по рейтингу и названию, чтобы наиболее привлекательные объекты отображались выше.

Получение объектов по массиву идентификаторов используется при построении маршрута.

```ts
const result = await query<AttractionRow>(
  "SELECT * FROM attractions WHERE id = ANY($1::uuid[])",
  [ids]
);

const byId = new Map(result.rows.map((row) => [row.id, toAttraction(row)]));
```

Здесь используется оператор `ANY`, позволяющий получить все объекты, идентификаторы которых входят в переданный массив. После получения данных создаётся `Map`, чтобы сохранить порядок объектов согласно порядку, выбранному пользователем.

## 4.7 Модуль пользовательской активности

Модуль пользовательской активности предназначен для фиксации действий пользователя. В текущей версии платформы сохраняются такие события, как просмотр маршрута, просмотр туристического объекта, поиск и создание маршрута.

Запись активности может использоваться для будущего развития рекомендательной системы. Например, если пользователь часто просматривает природные объекты, система может увеличить вес категории `nature`. Если пользователь регулярно строит короткие маршруты, можно автоматически подбирать компактные маршруты.

1. Пользователь выполняет действие в интерфейсе.
2. Клиент отправляет запрос на сервер.
3. Основной обработчик выполняет действие, например создаёт маршрут.
4. После успешного действия вызывается `trackActivity`.
5. В таблицу активности записывается событие.

В дальнейшем данный модуль можно использовать для аналитики, истории действий, улучшения рекомендаций и административной статистики.

## 4.8 Модуль расчёта расстояний

Для построения туристического маршрута необходимо рассчитать расстояние между точками. В системе для этого используется формула гаверсинусов, которая позволяет определить приблизительное расстояние между двумя координатами на поверхности Земли.

Функция расчёта принимает два туристических объекта и использует их широту и долготу.

```ts
export function haversineKm(a: Attraction, b: Attraction) {
  const dLat = toRad(b.latitude - a.latitude);
  const dLon = toRad(b.longitude - a.longitude);
  const lat1 = toRad(a.latitude);
  const lat2 = toRad(b.latitude);

  const value =
    Math.sin(dLat / 2) ** 2 +
    Math.sin(dLon / 2) ** 2 * Math.cos(lat1) * Math.cos(lat2);
```

Сначала разница широты и долготы переводится в радианы. Затем рассчитывается промежуточное значение `value`, которое используется в формуле гаверсинусов. Данный способ подходит для приближённой оценки расстояния между точками маршрута.

Финальный результат возвращается в километрах.

```ts
return 2 * EARTH_RADIUS_KM * Math.asin(Math.sqrt(value));
}

function toRad(value: number) {
  return (value * Math.PI) / 180;
}
```

Функция `toRad` переводит градусы в радианы, так как тригонометрические функции JavaScript работают с радианами. Значение `EARTH_RADIUS_KM` задаёт радиус Земли в километрах.

Расчёт общего маршрута выполняется как сумма расстояний между соседними точками.

```ts
const distance = points.reduce((sum, point, index) => {
  if (index === 0) return 0;
  return sum + haversineKm(points[index - 1], point);
}, 0);

const visitDuration = points.reduce(
  (sum, point) => sum + point.durationMinutes,
  0
);
```

Первый элемент маршрута не имеет предыдущей точки, поэтому для него расстояние не добавляется. Для остальных точек расстояние рассчитывается между текущей и предыдущей точкой. Также отдельно суммируется длительность посещения всех объектов.

## 4.9 Модуль оптимизации порядка точек

Пользователь может добавлять точки маршрута в произвольном порядке. Однако такой порядок не всегда удобен. Например, сначала может быть выбрана точка в центре города, затем точка на окраине, а потом снова объект в центре. Чтобы улучшить маршрут, используется простая эвристика ближайшего соседа.

Алгоритм начинает маршрут с первой выбранной точки и затем каждый раз выбирает ближайший ещё не посещённый объект.

```ts
  const optimized = [start];
  const queue = [...rest];

```

Если точек две или меньше, оптимизация не требуется. В остальных случаях первая точка сохраняется как начальная, а остальные помещаются в очередь. Далее алгоритм постепенно переносит точки из очереди в оптимизированный маршрут.

Выбор ближайшей точки выполняется внутри цикла.

```ts
const current = optimized[optimized.length - 1];
let nextIndex = 0;
let nextDistance = Number.POSITIVE_INFINITY;

queue.forEach((candidate, index) => {
  const distance = haversineKm(current, candidate);
  if (distance < nextDistance) {
    nextDistance = distance;
    nextIndex = index;
  }
});
```

Для каждой оставшейся точки рассчитывается расстояние от текущей последней точки маршрута. Если расстояние меньше текущего минимального значения, индекс этой точки сохраняется. После просмотра всех кандидатов ближайшая точка добавляется в маршрут.

Данный алгоритм не гарантирует математически оптимальный маршрут, но он прост, быстр и хорошо подходит для учебного проекта. Его преимущество заключается в понятной логике и небольшой вычислительной сложности, что важно для интерактивной работы пользователя с картой.

## 4.10 Модуль построения маршрута

Модуль построения маршрута объединяет несколько операций: проверку выбранных точек, загрузку туристических объектов из базы данных, оптимизацию порядка, расчёт общей дистанции и длительности. Он используется как для предварительного просмотра маршрута, так и для сохранения готового маршрута.

Предварительный просмотр маршрута нужен для того, чтобы пользователь мог увидеть результат до сохранения. Например, после выбора точек на карте система показывает примерное расстояние и время.

```ts
export async function buildRoutePreview(
  attractionIds: string[],
  shouldOptimize = true

  const attractions = await resolveAttractions(attractionIds);
  const points = shouldOptimize ? optimizeAttractions(attractions) : attractions;
  const summary = buildRouteSummary(points);

  return { points, summary };
}
```

На первом шаге функция получает объекты по их идентификаторам. Затем, если пользователь включил оптимизацию, вызывается `optimizeAttractions`. После этого рассчитывается сводка маршрута: общая дистанция и длительность.

Сохранение маршрута использует предварительный расчёт.

```ts
export async function saveRoute(input: {
  userId: string;
  name: string;
  attractionIds: string[];
  optimize?: boolean;
}) {
  const preview = await buildRoutePreview(input.attractionIds, input.optimize);
```

Такой подход исключает дублирование логики. Если предварительный просмотр и сохранение маршрута рассчитывались бы разными способами, пользователь мог бы увидеть одну дистанцию, а после сохранения получить другую. Использование общей функции делает поведение системы согласованным.

Перед расчётом маршрут проверяется.

```ts
if (attractionIds.length < 2) {
  throw new ApiError(400, "Для маршрута нужно выбрать минимум две точки");

const uniqueIds = Array.from(new Set(attractionIds));
const attractions = await findAttractionsByIds(uniqueIds);
```

Проверка запрещает строить маршрут из одной точки. Кроме того, с помощью `Set` удаляются повторяющиеся идентификаторы. Это защищает систему от некорректного ввода и случайных повторов при работе с интерфейсом.

## 4.11 Модуль сохранения маршрутов

После построения маршрута пользователь может сохранить его в личном профиле. Для этого используется модуль сохранения маршрутов. Он записывает общую информацию о маршруте и выбранные точки в разные таблицы базы данных.

Для сохранения используется транзакция, так как операция состоит из нескольких SQL-запросов. Сначала создаётся запись маршрута, затем сохраняются его точки. Если хотя бы один запрос завершится ошибкой, вся операция должна быть отменена.

```ts
const client = await pool.connect();

try {
  await client.query("BEGIN");
  const routeResult = await client.query<RouteRow>(
    `INSERT INTO routes (
       user_id, name, description, route_date, is_public,
       total_distance_km, total_duration_minutes
     )
```

В начале функция получает отдельное соединение из пула и запускает транзакцию командой `BEGIN`. После этого выполняется запрос на добавление маршрута. В запись включаются пользователь, название, описание, дата, признак публичности, общая дистанция и длительность.

После создания маршрута сохраняются его точки.

```ts
for (const [index, attraction] of input.points.entries()) {
  await client.query(
    `INSERT INTO route_points (route_id, attraction_id, position)
     VALUES ($1, $2, $3)`,
    [route.id, attraction.id, index + 1]
  );
}

await client.query("COMMIT");
```

Каждая точка получает позицию, которая определяет порядок посещения. После успешной записи всех точек выполняется `COMMIT`. Если в процессе произойдёт ошибка, управление перейдёт в блок `catch`.

```ts
} catch (error) {
  await client.query("ROLLBACK");
  throw error;
} finally {
  client.release();
}
```

Команда `ROLLBACK` отменяет все изменения, выполненные внутри транзакции. Блок `finally` освобождает соединение и возвращает его в пул. Это важно для стабильной работы сервера: если соединения не освобождать, со временем пул будет исчерпан.

Получение маршрута выполняется через объединение таблиц маршрутов, точек и туристических объектов.

```ts
const pointResult = await executor.query<RoutePointRow>(
  `SELECT
     rp.id AS point_id,
     rp.position,
     rp.note,
     rp.planned_start_time,
     a.*
   FROM route_points rp
   JOIN attractions a ON a.id = rp.attraction_id
```

Такой запрос возвращает не только идентификаторы точек, но и полную информацию о достопримечательностях. Благодаря этому клиентская часть может сразу отобразить название, категорию, координаты и описание каждой точки маршрута.

## 4.12 Модуль персональных рекомендаций

Модуль персональных рекомендаций является одним из ключевых элементов платформы. Его задача - помочь пользователю быстрее подобрать подходящие туристические объекты. В отличие от простого списка достопримечательностей, рекомендации учитывают данные профиля.

В текущей версии применяется балльная модель. Каждому объекту начисляются баллы по нескольким критериям:

- совпадение категории с интересами пользователя;
- совпадение тегов с интересами;
- соответствие бюджету;
- соответствие темпу путешествия;
- рейтинг объекта.

Основная функция рекомендаций сначала рассчитывает оценку каждого объекта.

```ts
const scored = attractions
  .map((attraction) => scoreAttraction(attraction, preferences))
  .sort((a, b) => b.score - a.score);

const routePoints = pickRoutePoints(scored, preferences);
const optimizedRoutePoints = optimizeAttractions(routePoints);
const summary = buildRouteSummary(optimizedRoutePoints);
```

После оценки объекты сортируются по убыванию балла. Далее из них выбираются точки будущего маршрута, порядок точек оптимизируется, а затем рассчитывается итоговая сводка.

Оценка интересов пользователя выполняется следующим образом.

```ts
const interestScore = preferences.interests.includes(attraction.category)
  ? 36
  : hasTagMatch(attraction, preferences.interests)
    ? 18
    : 0;
```

Если категория объекта прямо указана в интересах пользователя, объект получает 36 баллов. Если категория не совпала, но совпал один из тегов, объект получает 18 баллов. Если совпадений нет, дополнительная оценка по интересам не начисляется.

Бюджет учитывается через числовой ранг.

```ts
const budgetRank: Record<BudgetLevel, number> = {
  free: 0,
  low: 1,
  mid: 2,
  high: 3
};
```

Такое представление позволяет сравнивать уровни бюджета как числа. Например, если пользователь выбрал средний бюджет, ему подходят бесплатные, недорогие и средние объекты, но дорогие объекты получают штраф.

```ts
const budgetScore =
  budgetRank[attraction.budgetLevel] <= budgetRank[preferences.budget]
    ? 18
    : -10;
```

Если уровень бюджета объекта не превышает выбранный пользователем бюджет, объект получает прибавку. Если превышает, объект получает отрицательную оценку. Это не исключает объект полностью, но снижает вероятность его попадания в маршрут.

Темп путешествия влияет на выбор объектов по длительности посещения.

```ts
function getPaceScore(duration: number, pace: UserPreferences["pace"]) {
  if (pace === "calm") return duration <= 80 ? 14 : 0;
  if (pace === "active") return duration >= 60 ? 12 : 4;
  return duration <= 120 ? 10 : 2;
}
```

Для спокойного темпа предпочтительнее объекты с меньшей длительностью посещения. Для активного темпа допускаются более продолжительные объекты. Для сбалансированного темпа выбираются точки средней длительности.

Выбор точек для рекомендованного маршрута выполняется с ограничением по максимальной длительности.

```ts
for (const attraction of attractions) {
  const summary = buildRouteSummary([...selected, attraction]);
  if (
    selected.length < 2 ||
    summary.totalDurationMinutes <= preferences.maxDuration
  ) {
    selected.push(attraction);
  }
```

На каждом шаге система проверяет, не превысит ли добавление очередной точки максимальную длительность маршрута. При этом первые две точки добавляются в любом случае, чтобы сформировать минимально полноценный маршрут.

Данный модуль можно развивать дальше. Например, в будущих версиях можно учитывать историю просмотров, пользовательские оценки, сезонность, погоду, расстояние от текущего местоположения и популярность объектов среди других пользователей.

## 4.13 Модуль обработки HTTP-запросов

Серверная часть предоставляет клиенту набор HTTP-маршрутов. Каждый маршрут отвечает за отдельную группу операций: авторизацию, профиль, туристические объекты, маршруты и рекомендации.

Например, защищённые маршруты профиля сначала используют middleware авторизации.

```ts
router.use(authenticate);

router.get(
  "/",
  asyncHandler(async (req, res) => {
    const authReq = req as AuthenticatedRequest;
    const user = await findUserById(authReq.user.id);
```

Вызов `router.use(authenticate)` означает, что все последующие обработчики данного маршрутизатора требуют авторизации. Если пользователь не передал токен, обработчик профиля не будет выполнен.

Для обработки ошибок используется вспомогательная функция `asyncHandler`.

```ts
export function asyncHandler(
  handler: (req: Request, res: Response, next: NextFunction) => Promise<void>
) {
  return (req: Request, res: Response, next: NextFunction) => {
    handler(req, res, next).catch(next);
  };
}
```

В Express асинхронные ошибки необходимо передавать в `next`, чтобы они попали в общий обработчик ошибок. Функция `asyncHandler` избавляет от необходимости писать `try/catch` в каждом обработчике API.

Если возникает ошибка валидации, сервер возвращает понятный ответ клиенту.

```ts
if (error instanceof ZodError) {
  res.status(400).json({
    message: "Ошибка валидации данных",
    details: error.flatten()
  });
  return;
}
```

Такой ответ позволяет клиентской части показать пользователю, какие данные введены неправильно. Ошибки API также отделяются от непредвиденных внутренних ошибок, что делает поведение сервера более предсказуемым.

## 4.14 Модуль клиентского API

Клиентский API является прослойкой между React-компонентами и сервером. Он содержит функции для регистрации, входа, получения профиля, получения туристических объектов, рекомендаций и маршрутов. Благодаря этому компоненты не формируют HTTP-запросы вручную.

Токен доступа хранится в локальном хранилище браузера.

```ts
const TOKEN_KEY = "tourist_token";

export function getStoredToken() {
  return localStorage.getItem(TOKEN_KEY);
}

export function storeToken(token: string) {
  localStorage.setItem(TOKEN_KEY, token);
}
```

После успешной авторизации токен сохраняется через `storeToken`. При последующих запросах функция `request` извлекает токен и добавляет его в заголовок.

```ts
const headers: Record<string, string> = {
  "Content-Type": "application/json"
};

if (token) headers.Authorization = `Bearer ${token}`;
```

Таким образом, компоненты могут просто вызвать функцию `getProfile` или `saveRoute`, не думая о том, как именно передаётся авторизация. Это уменьшает повторение кода и упрощает поддержку.

Общая функция запроса обрабатывает HTTP-ответы.

```ts
if (!response.ok) {
  throw new Error(payload.message ?? "Ошибка запроса к серверу");
}

return payload as T;
```

Если сервер возвращает ошибку, она преобразуется в исключение. Компонент, вызвавший API-функцию, может перехватить это исключение и показать сообщение пользователю.

Модуль клиентского API выполняет следующие задачи:

- хранит и удаляет токен доступа;
- формирует HTTP-заголовки;
- отправляет JSON-запросы;
- обрабатывает успешные и ошибочные ответы;
- предоставляет компонентам удобные функции высокого уровня.

## 4.15 Модуль общего состояния приложения

Главный компонент приложения управляет основным состоянием клиентской части. В нём хранится текущий пользователь, выбранная вкладка, список туристических объектов, рекомендации, сохранённые маршруты, выбранные точки маршрута и данные предварительного расчёта.

Главный компонент также отвечает за сохранение маршрута.

## 4.16 Модуль авторизационного интерфейса

Модуль авторизационного интерфейса отвечает за отображение формы регистрации и входа. Пользователь может переключаться между двумя режимами: создание профиля и вход в существующий профиль.

Режим формы хранится в состоянии компонента.

```tsx
const [mode, setMode] = useState<"login" | "register">("register");
const [name, setName] = useState("");
```

## 4.20 Модуль форматирования пользовательских данных

В интерфейсе приложения числовые значения должны отображаться в понятном пользователю виде. Например, длительность маршрута хранится в минутах, но в интерфейсе удобнее показывать часы и минуты. Для этого используются небольшие вспомогательные функции форматирования.

Функция форматирования длительности преобразует минуты в строку.

```ts
export function formatDuration(minutes: number) {
  const hours = Math.floor(minutes / 60);
  const rest = minutes % 60;
  if (hours === 0) return `${rest} мин`;
  if (rest === 0) return `${hours} ч`;
  return `${hours} ч ${rest} мин`;
}
```

Если длительность меньше часа, отображаются только минуты. Если длительность равна целому количеству часов, минуты не выводятся. В остальных случаях показываются часы и оставшиеся минуты.

Форматирование расстояния выполняется отдельно.

```ts
export function formatDistance(km: number) {
  return `${km.toFixed(1)} км`;
}
```

Метод `toFixed(1)` округляет расстояние до одного знака после запятой. Это делает интерфейс аккуратнее: пользователю не нужно видеть слишком точные значения вроде `3.478231 км`.

Хотя данный модуль небольшой, он полезен для поддержания единого формата отображения данных во всех компонентах приложения.

## 4.22 Взаимодействие модулей при построении маршрута

Для понимания работы системы важно рассмотреть, как модули взаимодействуют между собой при построении маршрута. Данный сценарий является одним из основных в приложении.

Последовательность действий выглядит следующим образом:

1. Пользователь открывает карту.
2. Клиентская часть загружает список туристических объектов.
3. Пользователь выбирает несколько точек.
4. Главный компонент отправляет выбранные идентификаторы на сервер.
5. Сервер загружает объекты из базы данных.
6. Модуль маршрута проверяет корректность данных.
7. При необходимости выполняется оптимизация порядка точек.
8. Модуль расчёта расстояний вычисляет дистанцию и длительность.
9. Клиент отображает линию маршрута и итоговые показатели.
10. Пользователь сохраняет маршрут.
11. Сервер записывает маршрут и точки в базу данных.
12. Маршрут появляется во вкладке сохранённых маршрутов.

Данный сценарий показывает, что построение маршрута не является функцией одного компонента. В нём участвуют клиентские компоненты, клиентский API, серверные обработчики, сервис маршрутов, репозитории и база данных. Такое распределение обязанностей делает систему более гибкой.

## 4.23 Взаимодействие модулей при формировании рекомендаций

Формирование рекомендаций начинается с профиля пользователя. Пользователь задаёт интересы, бюджет, темп и максимальную длительность. Эти данные сохраняются на сервере и используются при каждом запросе рекомендаций.

Последовательность работы рекомендательной функции:

1. Клиент отправляет запрос на получение рекомендаций.
2. Сервер определяет пользователя по JWT-токену.
3. Загружаются данные профиля пользователя.
4. Загружается список доступных туристических объектов.
5. Для каждого объекта рассчитывается оценка.
6. Объекты сортируются по оценке.
7. Выбирается набор точек для маршрута.
8. Порядок точек оптимизируется.
9. Рассчитывается дистанция и длительность.
10. Клиент получает рекомендованные объекты и готовый маршрут.

Такой подход позволяет формировать рекомендации без сложных машинных моделей, но при этом учитывать индивидуальные настройки пользователя. Для дипломного проекта это решение является достаточно наглядным: можно объяснить, какие факторы влияют на результат и как они преобразуются в итоговую оценку.

## 4.24 Обработка ошибок и устойчивость системы

При разработке серверной части важно учитывать ошибки, которые могут возникнуть во время работы приложения. Пользователь может отправить некорректные данные, попытаться открыть чужой маршрут, передать неправильный токен или выбрать несуществующий туристический объект.

Для ошибок приложения используется класс `ApiError`.

```ts
export class ApiError extends Error {
  constructor(
    public statusCode: number,
    message: string
  ) {
    super(message);
  }
}
```

Данный класс хранит HTTP-код и текст ошибки. Это позволяет явно указывать, какая ошибка произошла: ошибка авторизации, ошибка валидации, отсутствие объекта или конфликт данных.

Например, если пользователь пытается построить маршрут из одной точки, сервер возвращает ошибку `400`.

```ts
if (attractionIds.length < 2) {
  throw new ApiError(400, "Для маршрута нужно выбрать минимум две точки");
}
```

Такой подход лучше, чем молчаливое игнорирование ошибки. Клиентская часть получает понятное сообщение и может показать его пользователю.

Общий обработчик ошибок скрывает внутренние детали сервера.

```ts
console.error(error);
res.status(500).json({ message: "Внутренняя ошибка сервера" });
```

Непредвиденные ошибки записываются в консоль сервера, но клиент получает нейтральное сообщение. Это защищает систему от раскрытия внутренних деталей реализации.

## 4.25 Модуль проектирования базы данных

Хотя база данных не является программным модулем в узком смысле, её структура напрямую влияет на реализацию серверных модулей. Поэтому при разработке платформы отдельно проектировались таблицы пользователей, туристических объектов, маршрутов, точек маршрута, отзывов и пользовательской активности.

Таблица пользователей хранит основные данные профиля и предпочтения в формате JSONB. Это удобно, так как предпочтения могут развиваться: к ним можно добавить новые поля без немедленного изменения структуры таблицы.

```sql
CREATE TABLE IF NOT EXISTS users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(120) NOT NULL,
  email VARCHAR(180) NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  role VARCHAR(30) NOT NULL DEFAULT 'traveler',
  avatar_url TEXT,
  preferences JSONB NOT NULL DEFAULT '{}'
);
```

Поле `id` имеет тип UUID, что удобно для распределённых веб-систем и снижает вероятность угадывания идентификаторов. Поле `email` имеет ограничение уникальности, так как один адрес электронной почты должен соответствовать только одной учётной записи. Пароль хранится в поле `password_hash`, что подчёркивает: в базу данных записывается не пароль, а результат криптографического хеширования.

Таблица туристических объектов содержит координаты и характеристики объекта.

```sql
CREATE TABLE IF NOT EXISTS attractions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(180) NOT NULL,
  city VARCHAR(120) NOT NULL DEFAULT 'Минск',
  category VARCHAR(80) NOT NULL,
  description TEXT NOT NULL,
  latitude NUMERIC(10, 6) NOT NULL,
  longitude NUMERIC(10, 6) NOT NULL
);
```

Координаты представлены отдельными полями `latitude` и `longitude`. Это позволяет использовать их при отображении маркеров на карте и при расчёте расстояний между точками. Категория объекта используется в рекомендательной системе для сопоставления с интересами пользователя.

Маршруты и точки маршрутов разделены на две таблицы. Такой подход соответствует нормализации данных: один маршрут может содержать много точек, поэтому связь реализуется через отдельную таблицу.

```sql
CREATE TABLE IF NOT EXISTS route_points (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  route_id UUID NOT NULL REFERENCES routes(id) ON DELETE CASCADE,
  attraction_id UUID NOT NULL REFERENCES attractions(id),
  position INTEGER NOT NULL,
  note TEXT,
  UNIQUE(route_id, position)
);
```

Поле `position` определяет порядок посещения точек. Ограничение `UNIQUE(route_id, position)` не позволяет двум точкам одного маршрута иметь одинаковую позицию. Связь `ON DELETE CASCADE` означает, что при удалении маршрута все его точки также будут удалены автоматически.

Для ускорения запросов добавлены индексы.

```sql
CREATE INDEX IF NOT EXISTS idx_attractions_category
  ON attractions(category);
CREATE INDEX IF NOT EXISTS idx_routes_user_id
  ON routes(user_id);
CREATE INDEX IF NOT EXISTS idx_route_points_route_id
  ON route_points(route_id);
```

Индекс по категории ускоряет фильтрацию туристических объектов. Индекс по пользователю ускоряет получение маршрутов конкретного пользователя. Индекс по `route_id` ускоряет загрузку точек выбранного маршрута.

Таким образом, структура базы данных поддерживает основные сценарии платформы: регистрацию, настройку профиля, просмотр объектов, сохранение маршрутов и анализ пользовательской активности.

Данные включают не только название и координаты, но и категорию. Категория `history` будет использоваться при сопоставлении с интересами пользователя. Если пользователь выбрал интерес "История", данный объект получит высокий балл при расчёте рекомендаций.

Теги позволяют сделать подбор более гибким.

```sql
  ARRAY['история', 'прогулка', 'центр']
)
ON CONFLICT (name, city) DO UPDATE SET
  image_url = EXCLUDED.image_url;
```

Конструкция `ON CONFLICT DO UPDATE` предотвращает повторное добавление одинаковых объектов при повторном запуске скрипта и одновременно обновляет данные уже существующих точек. Это удобно в процессе разработки: базу можно инициализировать несколько раз, не опасаясь появления дублей или устаревших изображений.

Начальное наполнение данных решает несколько задач:

- позволяет сразу проверить отображение карты;
- даёт материал для работы рекомендательной системы;
- упрощает тестирование поиска и фильтрации;
- позволяет демонстрировать сохранение маршрутов без ручного ввода объектов;
- создаёт основу для дальнейшего расширения каталога достопримечательностей.

В дальнейшем такой набор данных может быть заменён административным интерфейсом, через который администратор сможет добавлять новые туристические объекты без прямого доступа к базе данных.

## 4.27 Модуль валидации входных данных

Валидация данных является важной частью серверной разработки. Клиентская часть может ограничивать ввод пользователя, но сервер всё равно должен самостоятельно проверять все полученные данные. Это связано с тем, что HTTP-запрос может быть отправлен не только из разработанного интерфейса, но и из стороннего инструмента.

Для проверки данных используется библиотека Zod. Она позволяет описывать ожидаемую структуру объекта и автоматически получать ошибку, если данные не соответствуют требованиям.

Пример схемы создания маршрута представлен ниже.

```ts
const createRouteSchema = previewSchema.extend({
  name: z.string().trim().min(2),
  description: z.string().trim().optional(),
  routeDate: z.string().optional(),
  isPublic: z.boolean().optional()
});
```

Данная схема расширяет схему предварительного просмотра маршрута и добавляет поля, необходимые для сохранения. Название маршрута является обязательным и должно содержать минимум два символа. Описание, дата и признак публичности являются необязательными.

Проверка выполняется перед обращением к бизнес-логике.

```ts
const body = createRouteSchema.parse(req.body);
const route = await saveRoute({
  userId: authReq.user.id,
  ...body
});
```

Если данные некорректны, метод `parse` выбросит ошибку, и сохранение маршрута не начнётся. Это защищает сервисный слой от неправильных входных данных. Если данные корректны, они передаются в функцию сохранения маршрута вместе с идентификатором текущего пользователя.

Валидация используется не только для маршрутов, но и для регистрации, входа, профиля и фильтров туристических объектов. Такой подход делает API более надёжным и предсказуемым.

Преимущества выделения валидации:

- ошибки обнаруживаются до выполнения SQL-запросов;
- сервер возвращает понятные сообщения об ошибках;
- бизнес-логика получает данные ожидаемого формата;
- уменьшается вероятность некорректных записей в базе данных;
- упрощается сопровождение API при изменении структуры запросов.

## 4.28 Модуль удаления маршрутов

Помимо создания и просмотра маршрутов, пользователь должен иметь возможность удалить ненужный маршрут. Эта функция важна для управления личным кабинетом, так как со временем у пользователя может накопиться много маршрутов.

Удаление маршрута выполняется только для текущего пользователя. Это означает, что пользователь не может удалить маршрут, принадлежащий другой учётной записи.

```ts
export async function deleteRoute(routeId: string, userId: string) {
  const result = await query<RouteRow>(
    "DELETE FROM routes WHERE id = $1 AND user_id = $2 RETURNING *",
    [routeId, userId]
  );

  return (result.rowCount ?? 0) > 0;
}
```

В SQL-запросе используются два условия: идентификатор маршрута и идентификатор пользователя. Если маршрут существует, но принадлежит другому пользователю, удаление не произойдёт. Возвращаемое значение показывает, была ли реально удалена запись.

На уровне HTTP-запроса результат проверяется.

```ts
const deleted = await deleteRoute(String(req.params.id), authReq.user.id);
if (!deleted) throw new ApiError(404, "Маршрут не найден");
res.status(204).send();
```

Если маршрут не найден или не принадлежит пользователю, сервер возвращает ошибку `404`. При успешном удалении возвращается код `204`, который означает, что операция выполнена успешно и тело ответа не требуется.

В клиентской части после удаления маршрут убирается из состояния.

```tsx
await removeRoute(routeId);
setRoutes((current) => current.filter((route) => route.id !== routeId));
setNotice("Маршрут удалён");
```

Такой подход позволяет обновить интерфейс без повторной загрузки всех маршрутов. Пользователь сразу видит, что маршрут исчез из списка.

## 4.29 Модуль пользовательских уведомлений интерфейса

В клиентской части предусмотрены короткие уведомления о результате действий. Например, после сохранения маршрута пользователь видит сообщение о том, что маршрут сохранён в профиле. Если произошла ошибка, отображается сообщение об ошибке.

Для этого в главном компоненте хранятся состояния `notice` и `error`.

```tsx
const [notice, setNotice] = useState("");
const [error, setError] = useState("");

{(error || notice) && (
  <div className={error ? "toast error" : "toast"}>
    <span>{error || notice}</span>
```

Если есть сообщение об ошибке или успешном действии, отображается блок уведомления. Класс блока зависит от типа сообщения. Для ошибки используется дополнительный класс `error`, благодаря чему уведомление можно выделить цветом.

Закрытие уведомления выполняется кнопкой.

```tsx
onClick={() => {
  setError("");
  setNotice("");
}}
```

После нажатия оба состояния очищаются, и уведомление исчезает. Такой механизм делает интерфейс более понятным: пользователь получает обратную связь после действий, но уведомления не мешают дальнейшей работе.

Уведомления используются в следующих случаях:

- успешное сохранение маршрута;
- успешное удаление маршрута;
- обновление рекомендаций;
- обновление профиля;
- ошибка запроса к серверу;
- ошибка авторизации;
- ошибка сохранения маршрута.

Даже небольшой модуль уведомлений повышает удобство приложения, потому что пользователь понимает, завершилась ли операция успешно.

## 4.30 Модуль состояний загрузки

Веб-приложение взаимодействует с сервером асинхронно. Запросы к API занимают некоторое время, поэтому интерфейс должен показывать состояние загрузки и предотвращать повторные действия пользователя.

Для этого используется состояние `loading`.

```tsx
const [loading, setLoading] = useState(false);

async function refreshRecommendations() {
  setLoading(true);
  setError("");
  try {
    setRecommendations(await getRecommendations());
```

Перед началом запроса состояние `loading` устанавливается в `true`, а предыдущая ошибка очищается. После успешного ответа обновляются рекомендации. Если произойдёт ошибка, она будет обработана в блоке `catch`.

После завершения запроса загрузка отключается.

```tsx
} catch (caught) {
  setError(getErrorMessage(caught));
} finally {
  setLoading(false);
}
```

Блок `finally` выполняется всегда: и при успешном запросе, и при ошибке. Поэтому интерфейс не останется в состоянии бесконечной загрузки.

Состояние загрузки используется и в кнопках.

```tsx
<button
  className="primary-button"
  type="submit"
  disabled={selectedIds.length < 2 || loading}
>
```

Кнопка сохранения маршрута отключается, если выбрано меньше двух точек или уже выполняется запрос. Это защищает систему от повторной отправки одного и того же действия.

## 4.31 Модуль адаптивного интерфейса

Пользователь может открывать платформу на разных устройствах: ноутбуке, мониторе, планшете или смартфоне. Поэтому интерфейс должен подстраиваться под ширину экрана. Для этого используются CSS-правила с медиазапросами.

Например, при уменьшении ширины экрана меняется структура рабочего пространства.

```css
@media (max-width: 860px) {
  .planner-layout {
    grid-template-columns: 1fr;
  }

  .planner-sidebar {
    border-right: 0;
    border-bottom: 1px solid #dbe2ec;
  }
}
```

На широком экране список объектов и карта расположены рядом. На узком экране они перестраиваются в одну колонку. Это предотвращает сжатие карты и делает интерфейс пригодным для использования на мобильных устройствах.

Также отдельно задаётся высота карты для маленьких экранов.

```css
@media (max-width: 560px) {
  .map-surface,
  .map {
    min-height: 420px;
  }
}
```

Если не задать минимальную высоту, карта может стать слишком маленькой и неудобной для взаимодействия. Данное правило сохраняет достаточную рабочую область даже на смартфоне.

Адаптивный интерфейс важен для туристического приложения, так как пользователь может планировать маршрут не только дома, но и во время поездки с мобильного устройства.

## 4.32 Модуль расширяемости системы

При разработке платформы учитывалась возможность дальнейшего расширения. Система уже имеет задел для добавления новых функций без полной переработки архитектуры.

Например, таблица пользовательской активности может быть использована для более точных рекомендаций. Сейчас рекомендации строятся на основе предпочтений профиля, но в будущем можно добавить анализ фактического поведения пользователя.

```ts
await trackActivity({
  userId: authReq.user.id,
  activityType: "route_created",
  entityId: route.id,
  metadata: { points: route.points.length }
});
```

В данном фрагменте после создания маршрута записывается событие активности. В дополнительных данных сохраняется количество точек маршрута. Позже такие данные можно использовать для анализа того, какие маршруты чаще создаёт пользователь: короткие, длинные, активные или спокойные.

Также система может быть расширена административным модулем. В текущей версии туристические объекты добавляются через SQL-скрипт, но структура уже позволяет реализовать API для создания и редактирования объектов.

Возможные направления развития:

- добавление административной панели;
- загрузка изображений туристических объектов;
- система отзывов и оценок;
- построение маршрута по дорогам через внешний сервис;
- интеграция с погодным API;
- учёт текущего местоположения пользователя;
- совместное планирование поездки несколькими участниками;
- экспорт маршрута в PDF;
- добавление многоязычного интерфейса.

Благодаря модульной структуре такие функции можно добавлять постепенно. Например, интеграция с погодой потребует нового сервисного модуля, но не потребует переписывать модуль пользователей или сохранения маршрутов.

В ходе разработки программных модулей была реализована основная функциональность веб-платформы для подбора туристических маршрутов с персональными рекомендациями. Серверная часть обеспечивает авторизацию, работу с профилем, получение туристических объектов, построение маршрутов, сохранение маршрутов и расчёт рекомендаций. Клиентская часть предоставляет пользователю интерфейс регистрации, профиль, интерактивную карту, конструктор маршрута и список сохранённых маршрутов.
